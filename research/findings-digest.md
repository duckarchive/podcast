### Source 1 (quality: blog, date: 2026-06-09)
- **[supporting]** FCPXML is Apple's XML interchange format (.fcpxml) for Final Cut Pro that describes clips, projects, timelines, and metadata, allowing apps to exchange edits with Final Cut Pro.
- **[central]** FCPXML encodes timeline structure including clips, their source media, order and timing on the timeline, multiple video/audio lanes, transitions, effects, and markers.
- **[central]** DaVinci Resolve and Premiere Pro primarily read and write the legacy Final Cut Pro 7 XML (xmeml) format rather than the modern .fcpxml format, whose support varies.
- **[supporting]** Modern .fcpxml and legacy FCP7 xmeml are incompatible formats; sending modern .fcpxml to legacy-only applications will fail.

### Source 2 (quality: blog, date: 2011-09-20)
- **[central]** FCPXML expresses timing such as in/out points in time units (rational time, e.g. fractions of a second) rather than the frame units used by the legacy FCP7 XMEML format.
- **[central]** FCPXML and FCP7 XMEML are structurally incompatible formats and cannot be directly imported into the other application.
- **[supporting]** FCPXML organizes data with resources first and a single sequence containing storylines/clips in a parent-child relationship rather than the chronological browser/sequence/track structure of XMEML.
- **[supporting]** DaVinci Resolve supports both FCPXML and XMEML, but features that do not exist in the other application will not translate during round-trip interchange.

### Source 14 (quality: primary, date: ?)
- **[central]** Amazon Polly maps IPA/X-SAMPA phonemes to a compact set of single-character viseme codes for US English; the consonant set uses codes including p, t, S, T, f, k, i, l, r, s, u and vowels map to codes including @, a, e, E, i, o, O, u (the Oculus/Polly viseme scheme).
- **[central]** Multiple distinct phonemes collapse to the same viseme: bilabials /b/, /m/, /p/ all map to viseme 'p'; alveolars /d/, /n/, /t/ all map to viseme 't'; velars /g/, /h/, /k/, /ŋ/ all map to viseme 'k'; and postalveolars /dʒ/, /ʃ/, /tʃ/, /ʒ/ all map to viseme 'S'.
- **[supporting]** Polly's labiodental fricatives /f/ and /v/ both map to viseme 'f', and the dental fricatives /θ/ and /ð/ both map to viseme 'T', providing a many-to-one phoneme-to-viseme reduction usable for IPA-to-viseme mouth-shape mapping.
- **[tangential]** Stress and boundary symbols (primary stress ˈ, secondary stress ˌ, syllable boundary .) have no associated viseme (blank viseme column), meaning these markers should be ignored when generating mouth shapes.

### Source 18 (quality: primary, date: ?)
- **[central]** Amazon Polly defines a viseme as the visual equivalent of a phoneme, and provides phoneme-to-viseme mappings using IPA and X-SAMPA symbols, with per-language viseme sets — making it a usable phoneme-to-viseme mapping reference.
- **[central]** Polly's viseme labels are single/short letter codes (e.g. b->p, d->t, d͡ʒ->S, ð->T, f->f, g->k, h->k), grouping multiple phonemes into one viseme because some phonemes look identical when spoken — this is the Oculus-style viseme naming scheme rather than Preston Blair.
- **[central]** Polly returns viseme timing data as line-delimited JSON 'speech mark' objects, where each object has a 'time' field giving the timestamp in milliseconds from the start of the audio, a 'type' (including 'viseme'), and a 'value' holding the viseme name — directly usable for converting to frame timecodes.
- **[supporting]** Speech marks (including visemes) are intended for lip-sync use and are available across neural, long-form, and standard TTS engines.
- **[supporting]** Each language has its own viseme set tied to its phonemes, and Polly publishes complete per-language phoneme-to-viseme tables — but Ukrainian is not among Polly's supported voices, so Polly cannot directly generate visemes for Ukrainian audio.

### Source 19 (quality: primary, date: ?)
- **[central]** Cutlass generates FCPXML files targeting format version 1.13, using <fcpxml version="1.13"> as the root document container.
- **[central]** In FCPXML, still images (PNG/JPG) must be represented as Video elements with duration="0s" assets (formats omitting frameDuration), while videos use AssetClip elements with calculated duration and frameDuration; using the wrong element type crashes FCP.
- **[central]** FCPXML durations use a frame-aligned rational format with a 24000 timebase (e.g. "240240/24000s"), and decimal seconds must never be used because they cause sync drift.
- **[supporting]** Assets are defined with unique IDs (r1, r2, ...), file references via MediaRep using absolute file:// URIs, duration in frame format, and clips reference assets through ref attributes.
- **[supporting]** Cutlass's documentation makes no mention of DaVinci Resolve compatibility; it focuses exclusively on Final Cut Pro.

### Source 20 (quality: secondary, date: 2017)
- **[central]** FCPXML represents all time values (durations, offsets, start times) as rational numbers in the form numerator/denominator+'s', e.g. frameDuration of '2002/60000s' for 29.97 fps footage and '100/2500s' for 25 fps PAL, because rational time is more accurate than decimal frame rates.
- **[central]** Frame rate in FCPXML is the inverse of the frameDuration value, so a frameDuration of 2002/60000s yields 60000/2002 = 29.97002997 fps; this rational math governs seconds-to-frame timecode conversion.
- **[supporting]** Media files are declared once in a <resources> section with unique id attributes, and clips on the timeline reference them via a ref attribute (e.g. ref="r1") pointing to the corresponding <asset id="r1">.
- **[supporting]** FCPXML and the legacy FCP7 xmeml format are structurally incompatible: xmeml is track-based while FCPXML has no tracks, instead using a single primary storyline with connected clips, and effects/transitions are unlikely to transfer fully between them.
- **[supporting]** Time spans on clip elements are expressed with start and duration attributes both in rational-seconds form, as in a keyword spanning start="54263209/15000s" duration="33269236/60000s" within an asset-clip referencing ref="r1".

### Source 21 (quality: primary, date: 2017-03-27)
- **[central]** In FCP7 xmeml, a clip's timeline position is set by integer 'start' and 'end' subelements of clipitem (frame-based positions), distinct from 'in'/'out' which define media boundaries; start/end exist on clipitem but not on browser clip.
- **[central]** Still images (jpeg, tiff, PSD) are represented with stillframe=TRUE, and a stillframeoffset integer selects the freeze frame; notably, for still frames the duration value represents minutes rather than frames.
- **[supporting]** Frame rate is defined by a 'rate' element containing an integer 'timebase' plus a boolean 'ntsc'; setting ntsc=TRUE applies a 0.01% NTSC rate reduction to the timebase (e.g. timebase 30 + ntsc TRUE = 29.97 fps).
- **[supporting]** Timecode can be expressed as an integer 'frame' count (an actual frame count, not a time index, accounting for skipped numbers in drop-frame), with displayformat DF/NDF; e.g. string 01:00:2;05 at 29.97 fps equals frame count 107657 in drop frame.
- **[supporting]** Media files are referenced via a 'file' element using a 'pathurl' UTF-8 path that must be on a local volume and must start with file://localhost or file:///, with illegal characters percent-escaped per RFC 2396.

### Source 22 (quality: secondary, date: 2026-06-12)
- **[central]** steele-fcpxml is a Python library that programmatically generates Final Cut Pro XML (FCPXML) timelines from lists of clip specifications, producing files importable into DaVinci Resolve (or Final Cut Pro).
- **[supporting]** The library uses rational-time arithmetic with frame-rate detection so that timeline offsets do not drift, supporting standard frame rates via a FrameRate enum including 25, 29.97, and 23.976 fps.
- **[supporting]** The fluent API exposes add_clip() (with source in/out points), add_gap(), add_marker(), fork(), and write() to assemble and export .fcpxml files; a tc() helper parses MM:SS or HH:MM:SS timecodes into seconds.
- **[central]** The library does not explicitly document still-image or image-sequence clip support; only video clips are shown in examples, which is a limitation for placing many still mouth-shape images at exact frame timecodes.
- **[tangential]** steele-fcpxml requires Python 3.12+ and FFmpeg with ffprobe on PATH (for frame-rate detection), is licensed GPLv3+, and is in alpha development status.
